## 贯穿全场的主 Demo：AI 有声课件工坊

一个网页课件，由 Agent 自动完成完整流程：

```
教师输入主题（如"唐代诗歌的意象分析"）
  ↓
[Skill: 课件规划] 生成课件大纲（章节、知识点、时长）
  ↓
🔴 [Hook: 人工审核] 教师确认/修改大纲
  ↓
[Skill: 内容生成] 逐章节生成详细讲解内容
  ↓
[Skill: 测验生成] 根据内容生成 5 道选择题
  ↓
[MCP: TTS服务] 调用 TTS API 将每章摘要转成语音
  ↓
[Skill: 页面组装] 将所有内容组装成网页课件
  ↓
🔴 [Hook: 人工审核] 教师预览最终效果，确认或修改
  ↓
输出：完整的网页课件（文字 + 语音播放 + 测验）
```

**第 5 页：什么是 Agent（智能体）**

> Agent（智能体）= 能自主规划、执行多步骤任务、调用工具、在关键节点等待人工确认的 AI 系统。

**核心比喻**：

| 模型调用（上午） | Agent（下午） |
|---------|-------|
| 打电话问路："师傅，去故宫怎么走？" → 师傅告诉你 → 挂了 | 请了代驾："送我去故宫" → 司机自己导航、看路况、到地方停车 |
| 你问一句，它答一句 | 你定目标，它自己走，关键节点问你 |

---

**第 6 页：Agent vs 普通 AI 聊天的区别**

| 维度 | 普通 AI 聊天 | Agent（智能体） |
|------|-----------|----------------|
| 行为 | 你问一句，它答一句 | 你说一个目标，它自己规划并执行多个步骤 |
| 能力 | 只能生成文字 | 能读写文件、运行命令、搜索、调用外部 API |
| 主动性 | 被动回答 | 主动规划、主动发现问题 |
| 记忆 | 仅限当前对话 | 可以有长期记忆 |
| 比喻 | 百科全书 | 实习助理 |

---

**第 7 页：Agent 的工作循环 — LLM Loop**

```
       ┌─────────────────────────────────┐
       │                         │
       ▼                         │
   感知（读取文件、理解需求）         │
       │                         │
       ▼                         │
   推理（分析问题、制定计划）         │
       │                         │
       ▼                         │
   行动（修改文件、调用工具、执行命令）  │
       │                         │
       ▼                         │
   反馈（检查结果、发现新问题）──────────┘

   这个循环一直跑到任务完成。
```

> "这就是 Agent 和聊天机器人的本质区别——Agent 不是'答完就结束'，而是'做完才结束'。"

---

**第 8 页：Agent 的五个核心要素**

```
        ┌──────────────┐
        │    模型       │  ← 大脑：理解意图、做判断、生成内容
        │   (DeepSeek)  │
        └──────┬───────┘
               │
   ┌───────────┼───────────┐
   │           │           │
   ▼           ▼           ▼
┌──────┐  ┌──────┐  ┌──────────┐
│ 工具  │  │ 流程  │  │   记忆    │
│ (手)  │  │(步骤) │  │ (笔记本)  │
└──────┘  └──────┘  └──────────┘
               │
               ▼
        ┌──────────────┐
        │   人工审核     │  ← 把关：关键决策必须人确认
        └──────────────┘
```

---

**第 9 页：要素 ① 模型（大脑）+ ② 工具（手）**

- **模型**：负责"想"——DeepSeek 负责规划大纲、写内容、出题
- **工具**：负责"做"——读文件、写文件、调 API、执行命令
- 关键认知：**模型 ≠ Agent**。模型只决定"智力上限"，Agent 还需要工具、流程、记忆、审核

---

**第 10 页：要素 ③ 流程（步骤）+ ④ 记忆（笔记本）**

- **流程**：先做什么、后做什么、什么时候检查。课件工坊的流程有严格的顺序
- **记忆**：记住前面做了什么，出测验题时知道前面讲了什么知识点

---

**第 11 页：要素 ⑤ 人工审核（把关）**

- 关键决策不能交给 AI 自动完成
- 需要人工审核的场景：打分、发布给学生、学术内容、隐私和伦理
- Agent 的底线：**AI 干活，人把关**

---

**第 12 页：多智能体协作（拓展视野）**

- 当一个 Agent 不够时，多个 Agent 可以协作
- 三种模式：
  - **老板-员工模式**：一个领导 Agent 分配任务，多个执行 Agent 干活
  - **流水线模式**：需求 → 架构 → 编码 → 测试 → 部署，每个环节一个 Agent
  - **互审模式**：多个 Agent 互相检查工作

> "今天下午我们只学单 Agent——一个 Agent 已经能做很多事了。多 Agent 是明天下午李茂源和陈光老师的内容。"

### 1.2 Harness 理论+指令（14:35–15:00，25min）— 王钟声

**形式**：先 PPT 讲概念和指令，再现场演示验证。

#### PPT 内容（5 页）

**第 1 页：什么是 Harness**

> Harness = Agent 的执行框架 / "脚手架"

**三个比喻**：

| 比喻 | 对应 |
|------|------|
| 手机操作系统（iOS/Android） | 你感受不到它，但没有它 App 跑不起来 |
| 电影导演 | 不演戏，但决定谁上场、什么时候上场、怎么配合 |
| 工厂流水线传送带 | 把半成品送到下个工位，质检员在关键工位检查 |

> Claude Code 本身就是 Harness。你们上午已经用了，现在我们来认识它。

---

**第 2 页：Harness 的 7 层扩展体系**

```
⑦ Subagents（子代理）    ← 独立上下文并行干活
⑥ MCP Servers           ← 连接外部工具与数据源
⑤ LSP（语言服务器）      ← IDE 级代码导航
④ Plugins               ← Skills+Hooks+MCP 打包分发
③ Skills                ← 按需加载的专业知识包
② Hooks                 ← 会话生命周期钩子
① CLAUDE.md             ← 项目上下文文件
────────────────────────────
   模型本身（地板）
```

> **模型能力是地板，配置质量才是天花板。**
>
> 花时间把配置做好，比追最新模型版本更有实际收益。
>
> 初学者先把前三层做扎实：CLAUDE.md → Hooks → Skills。

---

**第 3 页：Harness 的 7 个核心职责**

| # | 职责 | 通俗解释 | 你在 Claude Code 里看到的表现 |
|---|------|---------|---------------------------|
| 1 | 接收任务 | 读懂你输入的目标描述 | 你的 prompt |
| 2 | 拆解步骤 | 把大目标分解成小任务 | Agent 的"思考"过程 |
| 3 | 调度工具 | 决定现在调用哪个工具 | 右侧的 `Read` `Write` `Bash` `API` |
| 4 | 传递上下文 | 把上一步结果喂给下一步 | Agent 不会"失忆" |
| 5 | 处理错误 | 调用失败时重试或换方案 | 自动重试、报错并继续 |
| 6 | 执行 Hook | 到审核点就停，等人操作 | "等待确认..." |
| 7 | 记录日志 | 每一步记下来，可回溯 | 对话历史、工具调用记录 |

---

**第 4 页：Harness 怎么"思考" — LLM Loop 机制**

```
用户输入任务
    ↓
Harness 分析 → 需要什么工具？
    ↓
调用工具 → 拿到结果
    ↓
分析结果 → 够了吗？还需要什么？
    ↓                    ↓
不够，继续调工具      够了，输出给用户
    ↓                    ↓
...循环...           等待用户反馈
```

> "这就是为什么 Claude Code 有时会连续调好几个工具——Harness 在做'思考→行动→观察→再思考'的循环。这个循环不停，直到任务完成。"

---

**第 5 页：Claude Code 中你需要知道的指令与配置**

| 指令/操作 | 作用 | 什么时候用 |
|----------|------|----------|
| 输入 prompt | 触发 Agent 开始工作 | 任何任务的开端 |
| 观察右侧工具调用日志 | 看 Harness 在调度什么工具 | 理解 Agent 在执行哪一步 |
| 回复"确认"/"继续" | 通过 Hook 审核点 | AI 停下来等你时 |
| 回复"修改：..." | 在 Hook 点修改后继续 | 对 AI 输出不满意时 |
| 把报错贴回去 | 让 Agent 自我修复 | 工具调用失败时 |
| `/skill <名称>` | 手动调用一个 Skill | 想用特定能力时 |
| `/model` | 切换模型 | 简单任务用便宜模型，复杂任务用强模型 |
| `settings.json` | 配置 Harness 行为（Hook、权限、MCP） | 定制 Agent 工作方式 |
| `CLAUDE.md` | 项目说明书，每次会话自动加载 | 让 AI 了解你的项目背景和规范 |

**三种权限模式**：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| Safe（安全） | 所有危险操作都需确认 | 新手阶段 |
| Approve（审批） | 大部分操作要确认 | 日常开发 |
| Edit（编辑） | AI 自主操作 | 完全信任 AI |

#### 现场演示（约 8min）

王钟声用课件工坊主题，演示 Harness 的实际调度过程：

1. 输入任务："帮我规划一个课件：主题'摄影构图基础'，面向大一新生，3 章。先生成大纲等我确认"
2. **逐步骤解说**工具调用日志：
   - "看右侧，Harness 先 Read 了 Skill 文件——它在准备"
   - "现在调用了模型 API——DeepSeek 在工作"
   - "大纲出来了——Harness 停了，因为你说'等我确认'——这是 Hook 在作用"
3. 教师回复"确认"，继续观察
4. "看，Harness 现在 Write 文件——在写内容"
5. "再调 API——生成下一章"
6. 强调：**整个过程不是你在操作，是 Harness 在自动调度**

### 1.3 Skill Part 1：使用现成 Skill（14:50–15:10，20min）— 林哲栋

**形式**：林哲栋演示 → 教师跟着操作。

#### Step 1：什么是 Skill（3min，口头讲）

> "回想 Agent 理论里'流程'那一页——好的流程应该封装起来，反复用。Skill 就是把一个任务的 SOP 封装好的'能力包'。"
>
> 比喻：Skill = 菜谱。你不会每次做红烧肉都重新想放什么——菜谱写好了，照着做就行。

**单次 Prompt vs Skill**：

| 维度 | 单次 Prompt | Skill |
|------|-----------|-------|
| 性质 | 一次性指令 | 可复用的标准流程 |
| 一致性 | 每次输出可能不同 | 每次都按同样标准 |
| 效率 | 每次重新写 | 一键触发 |
| 比喻 | 口头交代 | 书面 SOP |

#### Step 2：认识课件工坊的 4 个现成 Skill（5min，投屏展示）

展示 Skill 的目录结构：

```
.claude/skills/
├── courseware-planner/
│   └── SKILL.md          ← 课件规划
├── content-generator/
│   └── SKILL.md          ← 内容生成
├── quiz-generator/
│   └── SKILL.md          ← 测验生成
└── page-assembler/
    └── SKILL.md          ← 页面组装
```

> 一个 Skill 可以很简单（只有一个 SKILL.md），也可以很完整（SKILL.md + scripts/ + resources/ + references/）。我们今天用简单版本。

打开 `courseware-planner/SKILL.md`，讲解 Skill 文件的结构—— Frontmatter（name、description、trigger）+ 适用场景 + 输入要求 + 输出格式 + 执行步骤 + 示例 + 注意事项。

> "Skill 就是一个 Markdown 文件，不是代码。你写得越清楚，它执行越稳定。"

#### Step 3：教师动手 — 调用现成 Skill（12min）

教师在自己的 Claude Code 上操作：

1. **加载 Skill**：把 4 个 Skill 文件夹复制到 `.claude/skills/` 目录
2. **手动调用**：
   ```
   /skill 课件规划
   主题：____（每人选自己的主题），3章，面向大学生
   ```
3. **观察 Agent 输出**：Agent 生成结构化大纲
4. **先看看**：观察大纲结构是否合理
5. **继续让 Agent 跑**：回复"确认"，Agent 自动调用后续 Skill

> 林哲栋巡场，帮遇到问题的教师排错。强调："你现在不是在一问一答——Agent 在自动走完一个完整流程。"

### 1.4 Skill Part 2：自己写一个 Skill（15:10–15:30，20min）— 林哲栋

**形式**：林哲栋演示 → 教师动手创建。

#### Step 1：Skill 的写法 — 模板讲解（5min）

展示空白 Skill 模板：

```markdown
---
name: 你的Skill名称
description: 一句话描述这个Skill做什么
trigger: ["触发关键词1", "触发关键词2"]
---

# Skill名称

## 适用场景
什么时候应该用这个Skill？

## 输入要求
需要什么信息？

## 输出格式
输出应该长什么样？

## 执行步骤
1. 第一步
2. 第二步
3. ...

## 示例
一个具体的输入→输出示例

## 注意事项
有哪些坑要避免？
```

> "7 个部分，填完就是一个完整的 Skill。不需要写代码——用自然语言写就行。"

#### Step 2：现场示范 — 2 分钟创建一个 Skill（3min）

林哲栋对 Claude Code 说：
> "帮我创建一个 Skill，叫'课后作业生成'。根据课件内容生成 1 道开放式论述题。输入是全部章节内容，输出包含题目描述、参考答案要点（3-5个要点）、评分标准（4个维度）。"

Claude Code 自动生成 SKILL.md 文件。展示：
- 文件位置
- Frontmatter 已填充
- 7 个部分都已完成
- 不满意可以继续让 AI 修改

### 2.1 Hook 纯实践（15:50–16:20，30min）— 王钟声

**形式**：不讲 PPT，直接动手配、动手跑。

#### Step 1：1 分钟交代背景

> "刚才你们跑课件工坊的时候，Agent 在大纲生成后自动停了——那就是一个 Hook。现在我们要在自己创建的 Skill 流程里面，也加上这种'自动暂停等确认'的机制。"

#### Step 2：理解 Hook — 一张简图（3min）

```
Agent 工作流：
[步骤1] → [步骤2] → [Hook: 停！等人确认] → [步骤3] → [步骤4] → [Hook: 停！等人确认] → [输出]
```

> Hook 就是 AI 工作流中的"安检口"——到了关键节点，必须等人确认才能继续。

**三种 Hook 类型**：

| 类型 | 触发时机 | 举例 |
|------|---------|------|
| **PreToolUse** | 工具调用**之前** | AI 要写文件了，先问你"可以写吗？" |
| **PostToolUse** | 工具调用**之后** | AI 写完文件了，自动备份一份 |
| **Human-Review** | 关键决策点 | 大纲生成了、网页生成了，等你确认 |

#### Step 3：配置第一个 Hook（10min）

王钟声投屏演示，教师跟着操作：

1. 打开 `.claude/settings.json`
2. 加入 Hook 配置：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(git *)"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"AI 刚修改了文件，建议检查一下改了什么\""
          }
        ]
      }
    ]
  }
}
```

> "这段配置的意思是：每次 AI 写文件或编辑文件之后，自动弹一条提醒。这就是 PostToolUse Hook。"

3. 教师在自己的 settings 里加上
4. 重新跑课件工坊，观察效果：每次 Agent 写文件后，终端弹出提醒

#### Step 4：试试更强的 Hook — 加人工审核点（8min）

在 Claude Code 的任务描述中直接说明 Hook 点：

> "帮我做课件。先生成大纲。**大纲生成后必须停下等我确认**，我确认后才能继续写内容。内容写完后，**最终网页生成前也必须停下等我确认**。"

教师跑这个流程，体验 Agent 两次停下来等确认。

#### Step 5：在自己创建的 Skill 里加 Hook（8min）

任务：在刚才自己创建的 Skill 中，找合适的节点加上审核 Hook。

提示：
- "你的论文批改 Skill——在'给出评分'之前，要不要加 Hook？"
- "你的讨论题生成 Skill——在'发布给学生'之前，要不要加 Hook？"

王钟声巡场，引导思考：
> "Hook 放在哪里，取决于'哪一步做错了后果最严重'。后果越严重，越要靠前放 Hook。"

### 2.2 MCP 纯实践（16:20–16:50，30min）— 林哲栋

**形式**：不讲 PPT，直接配、直接跑。

#### Step 1：1 分钟交代背景

> "现在课件工坊生成的是纯文字网页。如果能把摘要自动转成语音，学生就能'听'课件了。怎么做？——通过 MCP 接入 TTS 语音合成服务。"

#### Step 2：理解 MCP — 一张简图（3min）

```
Claude Code（你的 AI 助手）
    │
    │ MCP 协议（统一的"插头标准"）
    │
    ├── TTS Server    → 微软语音合成 API → 文字转 MP3
    ├── 文件 Server    → 本地文件读写
    ├── 搜索 Server    → 搜索引擎，查最新资料
    └── （任何支持 MCP 的工具）
```

> "MCP（Model Context Protocol）= AI 世界的 USB Type-C。以前每连一个新工具都要单独写对接代码。MCP 一个标准全搞定。"

#### Step 3：MCP 与 Skill 的关系

- **Skill** 定义了"做什么、怎么做"（流程和规范）
- **MCP** 提供了"能力扩展"（让 AI 能做更多事）
- 两者互补：Skill 里可以调用 MCP 提供的能力

#### Step 4：配置 TTS MCP Server（5min）

林哲栋投屏，教师跟着操作：

1. 打开 `.claude/mcp.json`
2. 加入 TTS Server：

```json
{
  "mcpServers": {
    "edge-tts": {
      "command": "npx",
      "args": ["-y", "edge-tts-mcp"],
      "env": {
        "TTS_VOICE": "zh-CN-XiaoxiaoNeural"
      }
    }
  }
}
```

3. 保存，重启 Claude Code
4. 验证：问"你能调用 TTS 工具吗？"，确认连接成功

> "就这几行。AI 现在有了'嘴巴'——能把文字转成语音了。你不需要懂 TTS 的 API 细节——MCP Server 帮你封装好了。"

#### Step 5：跑完整流程（15min）

教师用自己选的主题，跑完整课件工坊流程（含 TTS 语音合成）：

1. 输入任务
2. Agent 规划大纲 → 教师确认（Hook #1）
3. Agent 生成内容 → 自动继续
4. Agent 调 TTS 转语音 → **观察 MCP 调用日志**
5. Agent 组装 HTML（含音频播放器）
6. 教师预览确认（Hook #2）
7. 浏览器打开网页 → 点击 ▶ 播放 → 听到 AI 生成的语音

> 林哲栋巡场，重点关注 MCP 调用是否成功。

#### Step 6：扩展思路（5min）

> "现在你们接入的是 TTS。但 MCP 能接的不止 TTS。"

快速展示其他 MCP Server：
- **搜索 Server**：Agent 可以上网查最新资料
- **翻译 Server**：课件自动生成双语版本
- **数据库 Server**：Agent 可以查询学生名单

> "关键不是今天接入多少工具——关键是理解了这个模式：有人写好 MCP Server → 你配置几行 → AI 就能用。这就是'即插即用'。"
